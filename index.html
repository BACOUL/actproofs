Why this exists

Modern systems are distributed and automated. Logs are internal, mutable, and often not designed as evidence. TimeProofs adds a neutral layer: publicly verifiable proof that an event occurred ‚Äî independent of the system that produced it.

Systems change

Infrastructure, vendors, and databases evolve. Evidence should survive migrations and failures.

Logs aren‚Äôt proof

Internal logs can be altered, truncated, or disputed. Proof must be verifiable by third parties.

Privacy matters

You should be able to prove an event without revealing the underlying content or personal data.

A timestamp in your system is not proof

A file ‚Äúcreation date‚Äù, a database row timestamp, or a screenshot is local and editable. A TimeProofs proof is verifiable: it proves when an exact event hash existed and that it has not changed.

TimeProofs does not certify intent, origin, legality, delivery, or success. It certifies two facts: existence at a time and integrity over time.

A concrete example

Scenario: automated email (or message)

A system generates an email via automation. Later, someone disputes whether it was issued at that time. With TimeProofs, you can keep a proof that the exact event (the message content hash + metadata hash) existed at timestamp T ‚Äî without revealing the message publicly.

Your organization can always verify internally by recomputing hashes from the original data. Third parties can verify the proof without accessing the underlying content.

How it works

Three steps: hash your event ‚Üí request a signed proof ‚Üí verify independently

1) Hash locally (SHA-256)

Your system computes a SHA-256 hash of the event payload (file, message, output, dataset snapshot, release manifest). The underlying data stays private.

2) Request a signed proof

Only a cryptographic hash is sent. The API returns a signed timestamp that becomes part of a proof file (.tproof.json). No server-side storage is required.

3) Store & verify later

Keep the proof file next to your records. Anyone can verify it (online or offline) to confirm the event existed at time T and remains unchanged.
